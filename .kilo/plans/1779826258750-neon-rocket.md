# Plan to Make visualfirst and archimate Plugins Installable in Turbo EA

## Overview

The goal is to make the `visualfirst` and `archimate` plugins installable separately in the Turbo EA application. Currently, the `archimate` plugin is partially integrated into the core codebase (with backend in `backend/app/plugins/archimate/` and frontend in `frontend/src/features/archimate/` and `archimate-lang/`), while the `visualfirst` plugin exists only in the frontend (`frontend/src/features/visualfirst/`). This plan outlines the steps to decouple both plugins from the core, place them in the `plugins` directory, and implement a dynamic plugin loading system so that they can be installed by simply copying their directories into `plugins/`.

## Current State

- **archimate**: 
  - Backend: `backend/app/plugins/archimate/` (seed, demo, migration, relationship checker, AMEFF)
  - Frontend: `frontend/src/features/archimate/` and `archimate-lang/`
  - Admin settings: Likely in `frontend/src/features/admin/ArchiMateAdmin.tsx` (inferred from grep)
  - Integrated in `backend/app/main.py` for seeding and migration (lines 611-622 for demo seeding, 624-635 for migration)
  - Routes likely hardcoded in frontend

- **visualfirst**:
  - Frontend: `frontend/src/features/visualfirst/`
  - No backend counterpart observed
  - No admin settings or seeding observed

- **plugins directory**: 
  - Already exists at `/plugins/`
  - Contains `archimate/` subdirectory with only a `README.md` file (the rest of the plugin is in the core)

## Plan

### Step 1: Design a Plugin System

We will implement a convention-based plugin system where each plugin resides in the `plugins/` directory as a subdirectory. Each plugin must follow a specific structure:

```
plugins/
  <plugin-name>/
    backend/          # Optional: Python backend code
      __init__.py
      plugin.py       # Entry point for backend initialization
      seed.py         # Optional: seeding functions
      seed_demo.py    # Optional: demo data seeding
      migrations/     # Optional: Alembic migrations specific to the plugin
    frontend/         # Optional: Frontend code
      src/
        features/     # Plugin-specific frontend components
        plugin-routes.ts  # Exports routes for the plugin
        admin/        # Optional: admin settings components (e.g., <PluginName>Admin.tsx)
    plugin.json       # Manifest: name, version, dependencies, etc.
```

The core application will:
- Scan `plugins/` for subdirectories at startup.
- For each plugin, if a `backend/` directory exists, dynamically import and initialize the backend.
- For each plugin, if a `frontend/` directory exists, dynamically load its routes and admin tabs.

### Step 2: Prepare the archimate Plugin for Installation

1. Move the existing archimate code from the core to `plugins/archimate/`:
    - Backend: Move `backend/app/plugins/archimate/` → `plugins/archimate/backend/`
    - Frontend: Move `frontend/src/features/archimate/` → `plugins/archimate/frontend/src/features/archimate/`
    - Langium: Move `archimate-lang/` → `plugins/archimate/archimate-lang/`
    - Ensure the backend maintains its internal structure (seed.py, seed_demo.py, migrate_unique.py, relationship_checker.py, ameff.py, etc.)

2. Create a backend entry point `plugins/archimate/backend/plugin.py` that exposes:
    - `seed_metamodel(db)`: seeds the ArchiMate metamodel
    - `seed_demo(db)`: seeds demo data (if enabled)
    - `run_migration(db)`: runs the unique migration (if enabled)

3. Create a frontend entry point `plugins/archimate/frontend/src/plugin-routes.ts` that exports the ArchiMate routes (e.g., for `/archimate` and `/archimate/:id/edit`).

4. Create an admin settings component `plugins/archimate/frontend/src/features/admin/ArchiMateAdmin.tsx` (if not already present) and export it via an index or manifest.

5. Add a `plugin.json` manifest:
    ```json
    {
      "name": "archimate",
      "version": "1.0.0",
      "description": "ArchiMate 3.2 plugin for Turbo EA",
      "backend": true,
      "frontend": true,
      "dependencies": []
    }
    ```

### Step 4: Backend Changes

Modify `backend/app/main.py` to:
1. Scan the `plugins/` directory for subdirectories.
2. For each plugin directory that contains a `backend/` subdirectory:
    - Dynamically add the plugin's `backend/` to the Python path.
    - Import the plugin's `plugin.py` module.
    - Call an initialization function (e.g., `init_app(app, db)`) if it exists, passing the FastAPI app and database session.
    - Alternatively, call specific functions like `seed_metamodel`, `seed_demo`, and `run_migration` based on environment variables or plugin settings.

    Example pseudo-code:
    ```python
    import os
    import importlib.util

    plugins_dir = os.path.join(os.path.dirname(__file__), "../../plugins")
    for plugin_name in os.listdir(plugins_dir):
        plugin_path = os.path.join(plugins_dir, plugin_name)
        backend_path = os.path.join(plugin_path, "backend")
        if os.path.isdir(backend_path):
            # Add backend_path to sys.path
            spec = importlib.util.spec_from_file_location(
                f"{plugin_name}_plugin",
                os.path.join(backend_path, "plugin.py")
            )
            plugin_module = importlib.util.module_from_spec(spec)
            spec.loader.exec_module(plugin_module)
            if hasattr(plugin_module, "init_app"):
                plugin_module.init_app(app, db)
    ```

3. Adjust the seeding and migration logic to be called from the plugin's `plugin.py` rather than hardcoded.
    - Specifically, in `backend/app/main.py`, replace the hardcoded calls to `seed_archimate_demo` and `migrate_archimate_unique` with calls to the plugin's functions.

### Step 5: Frontend Changes

Modify the frontend (likely in `frontend/src/App.tsx` and the admin settings page) to:
1. Fetch the list of available plugins from the backend (new endpoint: `GET /api/v1/plugins` that scans the `plugins` directory and returns plugin metadata).
    - Alternatively, during build time, scan the `plugins` directory and generate a routes configuration (but this requires a rebuild; runtime is better for true installability).

2. For each plugin that has a frontend:
    - Dynamically import the plugin's `plugin-routes.ts` (using dynamic import or federation) and add its routes to the router.
    - If the plugin provides an admin tab, dynamically import it and add it to the admin settings page.

    Since we are using Vite, we can use `import()` for code splitting.

    Example for routes:
    ```javascript
    const pluginRoutes = await import(`../../../plugins/${pluginName}/frontend/src/plugin-routes`);
    // Then add pluginRoutes.default to the router
    ```

3. Ensure that the plugin's assets (if any) are correctly served. This may require adjusting the Vite configuration to include the `plugins` directory in the asset handling.

### Step 6: Database Migration Handling

1. For each plugin that has migrations (in `backend/migrations/`), we need to integrate them with the main Alembic configuration.
    - Option 1: Copy the plugin's migration scripts into the main `backend/alembic/versions/` directory at install time (with a naming convention to avoid conflicts).
    - Option 2: Configure Alembic to scan multiple directories for migrations (including each plugin's `backend/migrations/`).

    We choose Option 2 for true pluggability. Modify the Alembic `env.py` to include migration directories from all plugins.

    Example in `backend/alembic/env.py`:
    ```python
    from sqlalchemy import engine_from_config
    from sqlalchemy import pool
    from logging.config import fileConfig
    import os
    import sys

    # Add plugin migration directories
    plugins_dir = os.path.join(os.path.dirname(__file__), "../../plugins")
    for plugin_name in os.listdir(plugins_dir):
        plugin_migrations = os.path.join(plugins_dir, plugin_name, "backend", "migrations")
        if os.path.isdir(plugin_migrations):
            sys.path.append(plugin_migrations)

    # ... rest of env.py
    ```

    Then, when running `alembic upgrade head`, it will pick up migrations from all plugins.

### Step 7: Seeding and Demo Data

1. Modify the seeding logic in `backend/app/main.py` to call the seeding functions from each plugin's `backend/plugin.py` (or `seed.py`/`seed_demo.py` if they exist) based on environment variables (e.g., `SEED_<PLUGIN_NAME>=true`).

    Example:
    ```python
    for plugin_name in enabled_plugins:  # determined by env vars
        plugin_module = importlib.import_module(f"plugins.{plugin_name}.backend.plugin")
        if hasattr(plugin_module, "seed_metamodel"):
            await plugin_module.seed_metamodel(db)
        if os.getenv(f"SEED_{plugin_name.upper()}") == "true":
            if hasattr(plugin_module, "seed_demo"):
                await plugin_module.seed_demo(db)
    ```

### Step 8: Admin Settings for Plugins

1. Create a new admin page or extend the existing settings page to list installed plugins and allow enabling/disabling them.
    - This page will read the `plugins/` directory and show each plugin's `plugin.json` metadata.
    - Provide toggle switches to enable/disable each plugin (storing the state in the database or in environment variables via a `.env` update mechanism).

2. When a plugin is enabled/disabled, the application may require a restart to load/unload the plugin. We can design the system to reload plugins without restart by:
    - For backend: Using a plugin manager that can unload and reload modules (complex due to Python's module caching).
    - For frontend: Since we are using dynamic imports, we can re-render the routes and admin tabs when the plugin enable state changes (without full reload).

    For simplicity, we can require a restart after enabling/disabling a plugin.

    - The visualfirst plugin works when placed in `plugins/visualfirst/` and enabled.
    - Disabling a plugin hides its routes and admin tabs, and does not load its backend.

2. Update the documentation to explain:

    - How to write a new plugin for Turbo EA.

## Risks and Considerations

- **Backend Plugin Loading**: Dynamically loading and unloading Python modules can lead to issues with module state and caching. We may need to use `importlib.reload` or design plugins to be stateless.
- **Frontend Bundle Size**: Loading plugins dynamically may increase bundle size if not code-split properly. We will use dynamic imports to load each plugin's code only when needed.
- **Database Migrations**: Ensuring that plugin migrations are applied in the correct order and do not conflict with core migrations.
- **Asset Handling**: Ensuring that plugin assets (images, etc.) are correctly served and cached.

## Conclusion

By following this plan, we will achieve a plugin system where both `visualfirst` and `archimate` can be installed separately by copying their directories into `plugins/` and enabling them via the admin settings (or environment variables). The core Turbo EA application will remain unchanged except for the addition of the plugin loading logic.
